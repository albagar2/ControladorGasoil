Sistema de Control de Gasoil Familiar

Versión 1.0 · Febrero 2026

🎯 ¿Qué es esta aplicación?

P: ¿De qué trata el proyecto?
R: Es una aplicación web full-stack diseñada para gestionar flotas de vehículos de forma familiar. Permite controlar vehículos, conductores, repostajes y mantenimientos, todo organizado por familias con autenticación segura mediante JWT.
P: ¿Qué tecnologías usa?
R: El frontend está construido con Angular 21, mientras que el backend usa Express.js + TypeORM + MySQL.

📂 ¿Cómo está organizado el proyecto?

P: ¿Cuál es la estructura de carpetas principal?
R: El proyecto tiene dos carpetas principales:
  • / (raíz) - Frontend Angular en el puerto 4200
  • /backend - API REST en el puerto 3001

Frontend (src/app/)

Carpeta Propósito
auth/ Login, registro, guards e interceptores
dashboard/ Panel principal con tabs
vehicles/ CRUD de vehículos
drivers/ CRUD de conductores
refuels/ Gestión de repostajes
maintenance/ Gestión de mantenimientos con filtros
family/ Administración familiar (solo admin)
services/ ApiService, MaintenanceService, ExportService

Backend (backend/src/)

Carpeta Contenido
controllers/ 6 controladores (auth, vehicle, driver, refuel, maintenance, family)
entities/ 5 entidades TypeORM (User, Vehicle, Driver, Refuel, Maintenance, Family)
middleware/ authMiddleware (JWT) y uploadMiddleware (Multer)
routes/ Definición de todas las rutas API
scripts/ seed.ts (datos de prueba)

🛣️ ¿Cómo funciona la navegación?

P: ¿Qué rutas públicas tiene la aplicación?
R: La aplicación tiene 3 rutas públicas principales:
  • / - Página de bienvenida
  • /login - Inicio de sesión

P: ¿Y las rutas protegidas?
R: Todo el contenido principal está en /dashboard, que usa tabs con query params:
/dashboard?tab=resumen → Vista general
/dashboard?tab=vehiculos → Gestión de vehículos
/dashboard?tab=conductores → Gestión de conductores
/dashboard?tab=repostaje → Gestión de repostajes
/dashboard?tab=mantenimiento → Gestión de mantenimientos
/dashboard?tab=familia → Admin de familias

Esto permite cambiar de vista sin recargar la página completa.

🔌 ¿Qué datos maneja la aplicación?

P: ¿Cuáles son las entidades principales?
R: La aplicación maneja 6 entidades principales con sus relaciones:

1. Vehicle (Vehículo)

{ id, matricula, modelo, combustible, anioMatriculacion, kilometrajeActual, propietarioId → Driver, familyId → Family, // Datos del seguro seguro_compañia, seguro_numero_poliza, seguro_fecha_vencimiento, seguro_cobertura, // Datos de ITV itv_estado, itv_fecha_caducidad, itv_kilometraje }

2. Driver (Conductor)

{ id, nombre, dni, email, telefono, fechaRenovacionCarnet, puntos, puntosMaximos, role: 'admin' | 'conductor', familyId → Family }

3. Refuel (Repostaje)

{ id, fecha, kilometraje, vehiculoId → Vehicle, conductorId → Driver, litros, precioPorLitro, costeTotal, proveedor, tipoCombustible, ticketImageUrl }

4. Maintenance (Mantenimiento)

{ id, fecha, kilometraje, vehiculoId → Vehicle, conductorId → Driver, tipo: 'Aceite' | 'Frenos' | 'Neumáticos' | ..., proveedor, costePieza, costeTaller, observaciones }

5. Family (Familia)

{ id, nombre, codigo // Código de invitación }

🌐 ¿Qué endpoints tiene la API?

Autenticación (sin protección)

Método Ruta Descripción
POST /api/auth/register Registrar usuario
POST /api/auth/login Iniciar sesión (devuelve JWT)

Recursos (protegidas con JWT)

P: ¿Todas las rutas de recursos siguen el mismo patrón?
R: Sí. Cada recurso (vehicles, drivers, refuels, maintenances) tiene exactamente 4 endpoints CRUD:
GET /api/{recurso} → Listar todos (filtrado por familia)
POST /api/{recurso} → Crear nuevo
PUT /api/{recurso}/:id → Actualizar existente
DELETE /api/{recurso}/:id → Eliminar
Por ejemplo: /api/vehicles, /api/drivers, /api/refuels, /api/maintenances

🔒 ¿Cómo funciona la seguridad?

P: ¿Qué sistema de autenticación usa?
R: La aplicación usa JWT (JSON Web Tokens). El flujo es:
  1. Usuario se registra → Password hasheado con bcrypt
  2. Usuario hace login → Backend genera JWT firmado
  3. Frontend guarda token en localStorage
  4. AuthInterceptor añade token en todas las peticiones: Authorization: Bearer <token>
  5. Backend verifica token en authMiddleware
P: ¿Cómo se protegen las rutas en el frontend?
R: Con AuthGuard (auth/auth.guard.ts). Si intentas acceder a /dashboard sin estar logueado, te redirige automáticamente a /login.
P: ¿Y en el backend?
R: Todas las rutas protegidas pasan por authMiddleware que:
  • Extrae el token del header
  • Verifica la firma con jwt.verify()
  • Decodifica el payload (userId, role, familyId)
  • Añade req.user con la información
  • Rechaza si token inválido o expirado

🗄️ ¿Cómo está configurada la base de datos?

P: ¿Qué motor de base de datos usa?
R: MySQL en el puerto 3306. La base de datos se llama vehiclemanagement.
P: ¿Cómo se crean las tablas?
R: TypeORM las crea automáticamente con synchronize: true en desarrollo. Las tablas se generan a partir de las entidades decoradas con @Entity().

⚠️ Importante: En producción debes desactivar synchronize y usar migraciones.

🚀 ¿Cómo ejecutar la aplicación?

Paso 1: Instalar dependencias

# Frontend cd controlGasoilFamiliar npm install # Backend cd backend npm install

Paso 2: Configurar base de datos

Asegúrate de tener MySQL corriendo y crea la base de datos:
CREATE DATABASE vehiclemanagement;
O ejecuta: npx ts-node src/scripts/createDatabase.ts

Paso 3: Ejecutar aplicación

# Frontend (puerto 4200) cd controlGasoilFamiliar npm run dev # Backend (puerto 3001) cd backend npm run dev

Paso 4: Cargar datos de prueba (opcional)

cd backend npx ts-node src/scripts/seed.ts
